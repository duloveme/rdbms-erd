# RDBMS React ERD Designer — 개발 계획서

## 1. 목표

- **라이브러리**: MS SQL, Oracle, MySQL, PostgreSQL을 우선 지원하는 React 기반 ERD 디자이너 패키지.
- **플레이그라운드**: Next.js 앱으로 라이브러리를 통합 검증·데모·문서화.

## 2. 범위 요약 (요구사항 대응)

| 구분 | 내용 |
|------|------|
| DB 우선순위 | MS SQL Server, Oracle, MySQL, PostgreSQL (방언·데이터 타입·식별자 규칙·DDL 차이 반영) |
| 논리/물리 | 논리 모델(개념명·도메인) ↔ 물리 모델(테이블·컬럼·타입) 전환 및 동기화 전략 |
| 편집 | Undo / Redo (`zundo` temporal 기반; §6.2) |
| 시각화 | 테이블별 색상 지정, 멀티 선택 정렬(왼쪽/가운데/오른쪽/위/중앙/아래/가로·세로 간격 동일) |
| 관계선 | 테이블 **우측 상단**에서 첫 엣지 시작; 연결 후 **아래쪽**에 추가 엣지 앵커; 엣지 드롭 시 **새 테이블 생성** 다이얼로그 |
| 산출물 | DDL, Index 생성 |
| JSON 입출력 | JSON은 기본적으로 **props**로 주입받아 현재 ER을 교체하고, UI의 **저장** 액션은 JSON payload 이벤트를 호스트에 전달한다. 또한 호스트는 `ref`의 `getJson()`으로 현재 JSON을 직접 조회할 수 있다 |
| Ref | React **`ref`**로 디자이너에 접근해 위 JSON API 및 선택적 명령(Undo/Redo, 포커스 등)을 **명령형**으로 호출 가능 |

## 3. 권장 모노레포 구조 (초안)

```
apps/
  playground/          # Next.js (App Router 권장)
packages/
  erd-designer/        # React 라이브러리 (캔버스, 상태, DDL 등)
  erd-core/            # (선택) DB 방언·AST·검증 순수 TS
```

- **플레이그라운드**는 라이브러리를 로컬 패키지로 링크하여 HMR로 빠른 피드백.
- **코어 분리**는 테스트·CLI 확장(후순)에 유리.

## 4. 기술 스택 제안

- **UI**: React 19, TypeScript.
- **다이어그램**: **React Flow(`@xyflow/react`) 확정**. 요구 UX(우측 상단 소스 핸들, 다중 핸들, 엣지 드롭 생성)에 맞춰 커스텀 노드/엣지를 구현한다.
- **상태**: **Zustand v5** + **`zustand/middleware/immer`**(중첩 모델 갱신). **Undo/Redo**는 **`zundo`**의 `temporal` 미들웨어로 구현한다(필요 시 레이아웃 전용 히스토리 분리 등은 `6.2` 정책에 맞춤). *Redux Toolkit은 채택하지 않음* — 컴포넌트 라이브러리에서 Provider·번들·소비 앱 전역 스토어와의 충돌을 피하기 위함.
- **폼/다이얼로그**: Radix UI + 프로젝트 스타일(플레이그라운드); 라이브러리는 headless 또는 최소 스타일로 제공해 소비 앱에 위임 가능.
- **DDL**: 방언별 **템플릿 + AST/모델** 이중화 권장(가독성·테스트·확장).

## 5. 도메인 모델 (초안)

- **Project**: 스키마 목록, 기본 방언.
- **Schema** (물리): 테이블, 뷰(후순), 관계, 인덱스.
- **LogicalEntity / LogicalAttribute**: 논리층; 물리 테이블·컬럼과 매핑(1:1 기본, N:1 분할 등은 후속).
- **Relationship**: cardinality, FK 컬럼 매핑, 삭제/갱신 규칙.
- **DiagramLayout**: 노드 위치, 줌, 레이어(선택); 모델과 분리해 직렬화 용이.
- **DesignDocument (JSON)**: 스키마·레이아웃 등을 하나의 문서 객체로 직렬화한다. 루트에 **`schemaVersion`**(또는 `version`)을 두어 이후 마이그레이션·호환성 검사에 쓴다.

## 6. 기능 상세 및 구현 메모

### 6.1 논리 ↔ 물리

- 전환 시 **이름 규칙**(스네이크/파스칼), **타입 매핑표**, **PK/FK 정책**을 설정으로 둠.
- **컬럼 최초 생성 규칙**: 컬럼 생성 시 논리 타입을 먼저 정하고, 물리 타입은 선택된 RDBMS 방언의 **기본 매핑 타입**으로 자동 설정한다.
- **물리 타입 개별 수정 허용**: 사용자는 물리 타입을 논리 타입과 독립적으로 직접 변경할 수 있다(예: `TEXT -> VARCHAR(200)`).
- **논리 타입 변경 시 재매핑**: 사용자가 논리 타입을 변경하면, 물리 타입은 해당 논리 타입에 대응하는 방언별 **일반 기본 타입으로 다시 설정**한다(수동 오버라이드 값은 재매핑으로 덮어씀).
- 충돌(이름 중복, 매핑 불가 타입)은 **진단 패널**로 수집.

### 6.2 Undo / Redo

- 기본 구현은 **`zundo`** `temporal` 스냅샷으로 두고, “무엇을 히스토리에 넣을지”는 **`partialize`** 등으로 조정한다.
- 사용자 의미 단위(추가/삭제/이동/속성 변경)가 한 번의 undo에 묶이도록 액션 설계를 맞춘다.
- 레이아웃만 바뀐 경우와 스키마 변경을 **히스토리에서 분리**할지(별 스토어·옵션) 정책을 정한다.

### 6.3 테이블 색상

- 테이블 노드 메타데이터에 `color` 또는 `themeToken`을 둔다. 이미지 내보내기 또는 호스트 JSON 기반 출력에서도 동일 색상을 반영한다.

### 6.4 멀티 선택 정렬 (레이아웃 편집)

- 멀티 선택된 테이블에 대해 정렬 명령을 제공한다: **왼쪽 맞춤, 가로 중앙 맞춤, 오른쪽 맞춤, 위쪽 맞춤, 세로 중앙 맞춤, 아래쪽 맞춤**.
- 분배 명령을 제공한다: **가로 간격 동일, 세로 간격 동일**.
- 기준점은 기본적으로 선택 그룹의 bounding box로 하고, 기준 노드(anchor) 기반 정렬은 후속 확장으로 둔다.
- 한 번의 정렬/분배 실행은 Undo 1스텝으로 기록한다(§6.2 정책 준수).

### 6.5 관계선 UX (요구사항 핵심)

- **Source handle**: 테이블 노드 우측 상단 고정.
- **다중 관계**: 기존 연결 수에 따라 **하단으로 순차 배치**되는 동적 핸들(또는 가상 핸들 + 엣지 라우팅).
- **엣지에서 드롭**: `onConnectEnd` / 커스텀 드래그 종료에서 빈 캔버스면 “새 테이블” 다이얼로그 → 생성 후 해당 엣지 완결.
- 라이브러리는 **이벤트 콜백**으로 다이얼로그 UI를 소비 앱에 맡기거나, 기본 내장 다이얼로그 제공 중 택일.

### 6.6 DDL / Index

- **DDL**: CREATE TABLE, ALTER(후순), 제약조건, 코멘트(방언별).
- **Index**: 컬럼 순서, 유니크, 포함 컬럼(SQL Server), 함수 기반 인덱스(Oracle/Postgres) 등 단계적.

### 6.7 디자이너 UI 구현 계획 (툴바·메뉴·대화상자·속성창)

컴포넌트 라이브러리 본체는 다이어그램 편집과 상태/모델에 집중하고, 제품 성격이 강한 UI는 호스트에서 구성 가능한 구조로 분리한다.

- **레이아웃**: `Toolbar`(상단) + `Canvas`(중앙) + `PropertyPanel`(우측) + `StatusBar`(하단, 선택) 4영역 기본.
- **툴바**: 모드 전환(논리/물리), Undo/Redo, 확대/축소, 자동정렬, **정렬/간격 맞춤**(왼쪽/가운데/오른쪽/위/중앙/아래/가로·세로 간격 동일), JSON **불러오기/저장** 버튼을 기본 슬롯으로 제공.
- **메뉴**: 파일(새로 만들기/불러오기), 편집(Undo/Redo), 보기(그리드/스냅), 모델(테이블/관계 추가)로 최소 메뉴만 정의.
- **대화상자**: `NewTableDialog`, `RelationshipDialog`, `ConfirmDialog`를 기본 제공하거나 콜백으로 완전 위임 가능한 하이브리드 구조를 채택.
- **속성창**: 선택 대상(테이블/컬럼/관계)에 따라 동적 폼을 렌더링하고, 변경은 단일 액션으로 상태 스토어에 반영(Undo 단위 보장).
- **확장 포인트**: 라이브러리는 `slots`/`render props`/콜백으로 호스트가 UI를 교체할 수 있게 하고, 플레이그라운드는 기본 구현체 데모를 제공.

### 6.8 JSON 직렬화 및 Ref(명령형 API)

부모 앱(플레이그라운드·포털·저장 서비스)이 디자인을 **파일·DB·API**와 주고받거나 자동화 테스트에서 상태를 주입하려면, 선언적 props만으로는 부족한 경우가 많다. 다음을 라이브러리 공개 계약에 포함한다.

#### JSON (props 기반)

- **입력(props)**: JSON(`DesignDocument`)은 컴포넌트 props로 주입받는다. 호스트가 새 JSON을 넘기면 현재 ER 상태를 **해당 JSON으로 교체**한다.
- **불러오기 동작**: UI에서 JSON 불러오기를 수행하면, 파싱/검증 후 기존 ER을 덮어쓰는 방식으로 동작한다(부분 병합 기본 비활성).
- **저장 동작(이벤트)**: 기존 “내보내기” 대신 **저장** 액션을 사용한다. 저장 클릭 시 현재 ER의 JSON payload를 이벤트(예: `onSaveJson`)로 호스트에 전달한다.
- **호환성**: 문서 루트의 **버전 필드**(`schemaVersion`)로 구버전 JSON 로드 시 마이그레이션 또는 명시적 거부를 할 수 있게 한다.

#### Ref

- 루트 디자이너 컴포넌트는 **`forwardRef`**와 **`useImperativeHandle`**(또는 동등한 안정적 패턴)으로 **핸들 타입**을 공개한다.
- 핸들에는 최소 `getJson()`을 포함해, 호스트가 필요 시 명령형으로 현재 JSON을 조회할 수 있게 한다.
- 선택 API(구현 시 확정): `undo()`, `redo()`, `focusTable(id)` 등. 로드/적용은 props 경로를 기본으로 유지한다.
- **Ref + props 원칙**: 상태 주입/교체는 props를 단일 소스로 하고, ref는 조회/명령형 보조 용도로 제한한다.

## 7. 추가로 검토하면 좋은 기능 (요구사항 보완)

우선순위는 제품 방향에 따라 조정.

1. **역공학(Import)**: 기존 DDL·DB 메타데이터에서 다이어그램 생성 (ERD 도구의 핵심 사용 시나리오).
2. **정합성 검증**: FK 타입 불일치, 순환 참조, 고아 엣지, PK 미지정 경고.
3. **비교·버전**: 모델 JSON 스냅샷 diff, 변경 요약(마이그레이션 메모용).
4. **코멘트/문서**: 테이블·컬럼 설명, 비즈니스 규칙 노트(보내기에 포함).
5. **네이밍 딕셔너리**: 접두어/접미어 규칙, 금지어 목록.
6. **다중 스키마·카탈로그**: PostgreSQL `schema`, SQL Server `schema`/`database`.
7. **뷰(View)**: 읽기 전용 노드 또는 점선 관계.
8. **접근성·국제화**: 키보드 조작, 스크린 리더 레이블, i18n.
9. **협업 포맷**: 모델 파일 단일 JSON/YAML 교환 표준.
10. **이미지/PDF보내기**: 리뷰·문서화.
11. **권한·민감도 태그**: PII 표시(문서화 목적, DDL 생성과 분리 가능).
12. **성능**: 대형 스키마(500 테이블 목표) 가상화·접기.

## 8. 비기능 요구사항

- **번들**: 라이브러리는 peerDependencies로 `react`, `react-dom` 명시.
- **테스트**: 코어 로직 단위 테스트(Vitest); 플레이그라운드 E2E(Playwright)는 주요 플로우만.
- **문서**: Storybook 또는 플레이그라운드 내 예제 페이지.
- **라이선스·기여**: OSS 시 CLA/라이선스 파일 정리.
- **JSON 스키마**: `DesignDocument`에 대한 TypeScript 타입(Zod 등으로 런타임 검증 선택)·**스키마 버전** 정책; breaking 변경 시 마이그레이션 가이드를 릴리스 노트에 남긴다.
- **공개 API**: Ref 핸들과 JSON 함수 시그니처는 semver 관점에서 **안정(stable)** 채널로 관리하고, 실험 API는 별도 네이밍 또는 플래그로 구분한다.

## 10. 결정사항 및 오픈 이슈

아래는 **확정된 결정과 남은 오픈 이슈**를 함께 정리한 섹션이다. 이미 확정된 항목은 구현 기준으로 사용하고, 미확정 항목은 논의 후 §4·§5·§6·§9 본문에 반영한다.

### 10.1 그래프 라이브러리 (결정 완료)

- **결정**: 다이어그램 엔진은 **React Flow(`@xyflow/react`)** 로 확정.
- **반영**: 노드/핸들/엣지 커스터마이징, `onConnectEnd` 기반 새 테이블 생성 플로우를 표준 구현으로 채택.

### 10.2 논리–물리 매핑 규칙 (결정 완료)

- **결정**: 논리명과 물리명은 **1:1 단순 매핑**으로 간다. 예: `사용자정보 -> STB_UserInfo`, `사용자아이디 -> UserID`.
- **결정**: 데이터 유형은 논리에서 일반화 타입(`TEXT`, `DATE`, `DATETIME`, `NUMBER`, `FLOAT`...)을 사용하고, 물리에서 선택한 RDBMS 방언 타입으로 변환한다.
- **반영**: 고급 상속/분할 모델링은 1차 범위에서 제외한다.

### 10.3 대규모 모델 성능 목표 (결정 완료)

- **결정**: 성능 기준은 **500개 테이블**을 목표로 한다.
- **기준**: 500개 테이블에서 기본 상호작용(패닝/줌/선택/드래그/속성편집)이 실사용 가능한 수준을 유지해야 한다.

### 10.4 `DesignDocument`에 담을 범위 (레이아웃 vs 전역 설정)

- **왜 필요한가**: JSON이 무엇을 포함하느냐에 따라 저장·diff·마이그레이션 전략이 달라진다.
- **권장안**: 최상위를 **`schemaVersion` + `model` + `layout`** 로 분리한다. `model`은 테이블·관계·인덱스·(논리 모드 시) 엔티티 등 **의미 있는 스키마**. `layout`은 노드 좌표·줌·선호 뷰포트. **네이밍 프리셋·테마** 등 전역 설정은 **`settings` (optional)** 블록으로 허용하되, 호스트가 무시해도 되도록 타입을 약하게 둔다. “레이아웃만 있는 파일”은 `model` 빈 객체로 표현 가능하게 할지 여부는 **불가(항상 최소 빈 스키마)** 권장 — 파서 단순화.
- **합의 시 산출**: `DesignDocument` 타입 정의 초안 + 예시 JSON 1개.

### 10.5 Ref 핸들 — 동기 vs 비동기 로드(대형 모델, 500테이블 기준)

- **왜 필요한가**: 500 테이블 규모 JSON에서 메인 스레드 블로킹·UX 멈춤이 날 수 있다.
- **권장안**: **MVP는 동기** — `setDesign(doc)` / `loadDesignJson`은 파싱+검증+적용을 한 틱에 끝낸다. 대신 문서에 **권장 상한**(예: 테이블 500개 또는 JSON n MB)을 적고 초과 시 호스트에 **청크 분할·서버 전처리**를 권장. **비동기 API**(`loadDesignAsync` + 진행 콜백 또는 Worker)는 상한 초과 사례가 실제로 나온 뒤 **마이너 버전**으로 추가.
- **합의 시 산출**: 공개 API 시그니처 + 상한·에러 타입 목록.
